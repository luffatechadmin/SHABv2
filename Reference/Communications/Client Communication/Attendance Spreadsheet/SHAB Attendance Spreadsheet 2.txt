## Attendance System Design – Conceptual Explanation (For Development)

The attendance system we are building is not just a simple log viewer, but a structured system that transforms raw scan data into meaningful, readable, and actionable information. The core challenge is balancing **data completeness** with **usability**, especially when dealing with many fields such as multiple scan times, calculated hours, and behavioral indicators. If all of these are displayed at once in a table, it will inevitably lead to horizontal scrolling, which significantly reduces usability and clarity.

To solve this, the system should be designed with a clear separation between **data storage** and **data presentation**. From a data perspective, we will maintain three logical levels: daily, weekly, and monthly. The daily level captures detailed operational data such as scan times, calculated hours, and attendance anomalies. The weekly level aggregates this data to reveal patterns such as consistency, punctuality, and frequency of issues. The monthly level further summarizes the information into metrics suitable for payroll processing and performance evaluation. This layered structure ensures that each level serves a distinct purpose while remaining connected, where daily data feeds into weekly summaries, and weekly data feeds into monthly summaries.

However, the main design consideration lies in how this data is presented to the user. Instead of displaying every available field in a single wide table, the interface should follow a principle of **progressive disclosure**. This means that only the most essential information is shown in the default view, while additional details are revealed only when needed. For example, in the daily table, the default view should include only key fields such as employee name, date, shift, first check-in, last check-out, total worked hours, overtime, attendance status, and a summarized flags column. This keeps the table compact and readable without requiring horizontal scrolling.

More detailed information, such as multiple scan entries (first in/out, second in/out, overtime in/out), break durations, and behavioral metrics like lateness or early leave, should not be displayed in the main table. Instead, these should be accessible through an **expandable row** or a secondary detail view when the user interacts with a specific record. This approach ensures that detailed data remains available without overwhelming the primary interface.

A key simplification technique in this design is the consolidation of multiple related fields into a single summarized column. Specifically, various data quality indicators—such as missing scans, duplicate scans, and odd-time scans—should not be displayed as separate columns. Instead, they should be combined into a single “Flags” field that summarizes all issues in a concise, human-readable format (for example, “MISSING_OUT | DUPLICATE x2 | ODD”). This significantly reduces the number of columns while preserving all necessary information.

The same principle applies to weekly and monthly tables. These tables are inherently more summarized and should remain even more compact. The default view should focus only on key metrics such as total worked hours, attendance counts, and performance indicators like attendance rate and punctuality. Additional breakdowns, such as counts of late arrivals or anomaly occurrences, can be accessed through optional expanded views or toggles. This ensures that higher-level summaries remain clean and decision-friendly.

Another important aspect of the design is the use of **derived data** instead of storing redundant fields. For example, the “last out” time does not need to be stored explicitly if it can be determined from the latest scan record. Similarly, the flags summary can be generated dynamically based on underlying conditions. This reduces data duplication, minimizes inconsistencies, and keeps the system more maintainable.

It is also important to emphasize that we should not attempt to solve the wide-table problem by splitting the data into multiple database tables purely for display purposes. Doing so would introduce unnecessary complexity and potential data integrity issues. Instead, the correct approach is to maintain a clean and normalized data structure in the backend while handling complexity at the presentation layer through grouping, summarization, and interaction design.

In summary, the system should be built on three key principles: **completeness in data, simplicity in presentation, and flexibility in interaction**. All relevant attendance data must be captured and stored, but only the most important information should be displayed by default. Additional details should be revealed progressively through user interaction. This approach ensures that the system remains both powerful and user-friendly, avoiding clutter while still supporting detailed analysis and accurate payroll processing.
