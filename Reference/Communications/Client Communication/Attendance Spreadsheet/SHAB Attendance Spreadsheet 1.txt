Attendance System – Table Design & UI Approach (Developer Note)
Objective
We need to design an attendance system that:
•	Tracks employee attendance accurately
•	Supports payroll and performance analysis
•	Avoids wide tables and horizontal scrolling
•	Remains scalable and easy to maintain
________________________________________
1. Overall Approach
We will build 3 levels of data tables:
1.	Daily (detailed, operational)
2.	Weekly (pattern & trends)
3.	Monthly (summary & decision-making)
⚠️ Important:
These are logical layers, not necessarily separate UI tables shown at once.
________________________________________
2. Key Design Principle (Very Important)
Do NOT display all data fields at once.
Instead:
•	Keep the main table compact
•	Show additional details via:
o	Row expansion
o	Toggle views (tabs/buttons)
o	Grouped fields
Goal:
👉 No horizontal scrolling in main view
________________________________________
3. DAILY TABLE DESIGN
This is the most detailed dataset.
________________________________________
3.1 Required Data (Backend Structure)
Each record should include:
Core Info
•	staff_id
•	name
•	date
•	shift_assigned
Raw Time Points
•	first_in
•	first_out
•	second_in
•	second_out
•	ot_in
•	ot_out
Calculated Metrics
•	total_worked_hours
•	break_duration
•	scheduled_hours
•	overtime_hours
Behavior Metrics
•	late_minutes
•	early_leave_minutes
Data Flags (Boolean or Count)
•	missing_in
•	missing_out
•	duplicate_scans
•	odd_time_scan
Status
•	attendance_status
________________________________________
3.2 UI Display (IMPORTANT)
Default View (Compact – No Scroll)
Only show:
•	Name
•	Date
•	Shift
•	First In
•	Last Out (derived from latest scan)
•	Total Hours
•	OT Hours
•	Status
•	Flags (combined)
________________________________________
3.3 Flags Column (Replace Multiple Columns)
Instead of showing multiple columns:
•	missing_in
•	missing_out
•	duplicate_scans
•	odd_time_scan
Combine into ONE field:
Example:
MISSING_OUT | DUPLICATE x2 | ODD
________________________________________
3.4 Row Expansion (Click to View Details)
When a row is clicked, show:
Time Breakdown
•	first_in / first_out
•	second_in / second_out
•	ot_in / ot_out
Additional Info
•	break_duration
•	late_minutes
•	early_leave_minutes
________________________________________

________________________________________
6. Important Implementation Notes
6.1 Avoid Horizontal Scrolling
•	Max ~8–10 columns in default view
•	Everything else goes into expandable sections
________________________________________
6.2 Derived Fields (Do NOT store redundantly)
•	last_out = latest scan
•	flags = generated dynamically
________________________________________
6.3 Reusability
•	Weekly data should be computed from Daily
•	Monthly data should be computed from Weekly (or Daily aggregate)
________________________________________
6.4 UI Suggestions (Recommended)
•	Use badges/tags for flags:
o	Red → critical (missing scan)
o	Yellow → warning (late)
o	Blue → info (OT)
________________________________________
7. Summary
We are designing:
•	A clean data structure (3 levels)
•	A compact UI (no horizontal scroll)
•	A flexible system (expandable details)
Key idea:
👉 Show less by default, reveal more on demand
