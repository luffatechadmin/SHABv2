

The purpose of this enhancement is to evolve the current attendance system from a single-device integration into a scalable, multi-device ingestion platform capable of handling different hardware behaviors and data formats. At present, the system is tightly coupled to the ZKTeco WL10 device, which operates using a predictable structure in both its data format and file handling mechanism. This includes a fixed attendance log filename (`1_attlog.dat`) and a consistent column structure, allowing the existing middleware to parse and process attendance data reliably. However, with the introduction of the NGTeco W30 device, the system must now accommodate a fundamentally different behavior in both file generation and data structure. 

Unlike the WL10, the W30 device generates attendance logs using dynamically created filenames that include timestamps, such as `AttendanceLog04192026061238.dat`. This means that the system can no longer rely on a fixed filename or assume that only one file exists at any given time. Instead, multiple files may accumulate over time, each representing a batch of attendance data. This shift introduces a critical architectural requirement: the system must become independent of filename assumptions and instead operate based on file discovery and tracking logic. 

To address this, the system must implement a production-grade file handling strategy that processes all new files rather than targeting a specific file. The core idea is to treat the file system as a stream of incoming data sources, where each file is processed exactly once. This requires the middleware to scan a designated directory for all relevant attendance log files, identify which files have not yet been processed, and then process only those new files. After processing, each file must be marked as completed to prevent reprocessing in future cycles. This approach ensures that dynamically generated files from devices like the W30 are fully supported without requiring manual intervention or fragile filename matching logic. 

An important aspect of this design is the introduction of a tracking mechanism for processed files. The system must maintain a record—either in a database table or local state—of all files that have already been handled. This allows the middleware to distinguish between new and previously processed files during each scan cycle. By doing so, the system achieves idempotency, meaning it can run repeatedly without creating duplicate data or reprocessing the same input. This is especially important in environments where the middleware runs continuously or on a schedule, as it guarantees stability and consistency in data ingestion. 

In addition to file-level tracking, the system must also retain its existing record-level deduplication logic. Since attendance devices often return overlapping or repeated data, the database must continue enforcing uniqueness constraints (such as a combination of staff ID, timestamp, and device ID). This dual-layer approach—file-level deduplication combined with record-level deduplication—ensures that the system remains robust even if files are accidentally duplicated, re-imported, or partially processed. 

This enhancement must also integrate seamlessly with the broader requirement of supporting multiple device types. Each device may differ not only in file naming behavior but also in data structure and semantics. Therefore, while the file handling mechanism determines which files to process, the parsing logic must remain device-aware, selecting the appropriate parser based on the device type (e.g., WL10 vs W30). Regardless of these differences, all parsed data must be normalized into a single unified format before being stored or displayed. This ensures that the rest of the system—including the database schema and frontend dashboard—remains unchanged and continues to function consistently. 

From a design perspective, this enhancement represents a shift away from rigid, assumption-based logic toward a more flexible and extensible architecture. The system must no longer assume that devices behave uniformly, that filenames are stable, or that only one file exists per device. Instead, it must be built to accommodate variability in device behavior, making it resilient to differences in firmware, data export methods, and future device integrations. This approach not only solves the immediate challenge of supporting the W30 device but also lays the foundation for integrating additional devices in the future without requiring significant rework. 

In summary, the intention of this enhancement is to create a robust file ingestion layer that can handle both static and dynamically generated attendance log files by processing only new files, tracking what has already been processed, and preventing duplication at both file and data levels. This ensures reliable, automated, and scalable attendance data integration across multiple devices, while preserving the integrity and consistency of the overall system.
