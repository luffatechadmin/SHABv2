In addition to acting as a bridge between the fingerprint attendance device and the backend system, the middleware is intended to function as a local control center and monitoring dashboard for the attendance device. This means the middleware should not operate purely as a background service, but instead provide a user interface that allows administrators to view, manage, and configure how data is retrieved, processed, and synchronized with the central database (Supabase).

The fingerprint device (WL10) communicates over a TCP protocol and does not provide built-in visibility into its connection status or data flow beyond its small hardware screen. Therefore, the middleware dashboard will serve as a transparent layer that exposes the device’s behavior and data in a more accessible and user-friendly way. When the middleware connects to the device, it should display essential connection details such as the device’s current IP address, port, connection status (connected/disconnected), and last successful sync time. This is especially important because the device may be moved between networks or assigned a different IP address via DHCP. The dashboard should make it easy for an administrator to quickly identify and verify the device’s network configuration without needing to access router settings or the device menu manually.

A key feature of this middleware dashboard is the ability to display parsed attendance data locally in real time or near real time. As the middleware retrieves logs from the device, it should present both the raw data (e.g., user ID, timestamp, verification type) and processed or structured data in a readable format. This allows administrators to immediately confirm whether the device is functioning correctly and whether attendance records are being captured as expected. In addition to raw logs, the dashboard should include basic analytics such as total check-ins for the day, unique employees present, recent activity feed, and possibly simple visual summaries like counts per hour. These insights provide quick operational visibility without requiring access to the full web application.

The dashboard should also include manual and automated synchronization controls. Administrators should be able to configure the polling interval (for example, every 5 seconds, 10 seconds, or 1 minute) depending on how real-time they want the system to behave and how much network load is acceptable. There should also be a clearly visible button to trigger an immediate manual sync, which forces the middleware to connect to the device and fetch the latest attendance logs on demand. This is useful for troubleshooting, testing, or ensuring data is up to date without waiting for the next scheduled interval.

Another important function of the dashboard is to manage how data is sent to the central backend, which in this case is Supabase. The middleware should allow configuration of the API endpoint or Supabase credentials, and provide status feedback on whether data syncing is successful. For example, it can show the number of records successfully pushed, failed attempts, and retry status. This ensures that administrators can monitor not only device connectivity but also data integrity across the entire pipeline from device to cloud database.

From a usability standpoint, the middleware dashboard should be lightweight and accessible via a browser on the local network (for example, running on http://localhost:port or a local IP address). It should be designed with clarity in mind, offering sections such as device connection status, live attendance logs, analytics summary, and synchronization settings. The goal is to give administrators a single place to manage the entire attendance data flow without needing technical knowledge of the underlying protocol.

In summary, the middleware is not only a data bridge but also an interactive local dashboard that provides visibility, control, and configurability over the attendance system. It enables administrators to monitor device connectivity, inspect real-time attendance data, adjust synchronization behavior, and ensure reliable data transfer to Supabase. This approach significantly improves maintainability and usability, especially in environments where the device may be relocated, reconfigured, or operated by non-technical staff.
