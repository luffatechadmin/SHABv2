We are developing a web-based employee attendance system using a React and Tailwind CSS frontend, supported by a backend API and database. The objective of this system is to record, store, and display employee attendance data in near real-time when employees clock in and out using a fingerprint attendance device. The expectation is that attendance records will be reflected in the web application with minimal delay and presented through a live dashboard interface.

However, the fingerprint device we are using operates as a standalone SSR (Self-Service Recorder) terminal and does not support modern web integration features such as HTTP APIs or ADMS (push-based communication). Instead, it communicates using a proprietary TCP protocol over port 4370 and stores attendance logs internally. This means the device does not actively send attendance data to external systems, and therefore our web application cannot receive data directly from it. Because of this limitation, we must implement an intermediary layer to retrieve and forward the data.

To address this, the system will include a middleware service that acts as a bridge between the fingerprint device and our backend. This middleware will run on a machine within the same local network as the device (such as an office PC or dedicated server) and will connect to the device over TCP/IP using its IP address and port 4370. Using a compatible library or protocol implementation, the middleware will periodically query the device for attendance logs. Since the device does not track incremental updates, it typically returns all stored logs on each request, so the middleware must handle repeated data carefully.

Once the attendance data is retrieved, the middleware will transform it into a structured JSON format and send it to our backend API via HTTP requests. The backend will then process and store the data in a database. Because the device may repeatedly return the same attendance records, the backend must implement a deduplication mechanism, typically by enforcing a unique constraint on a combination of employee ID and timestamp. This ensures that duplicate entries are not stored even if the same data is received multiple times.

From there, the frontend application built with React and Tailwind will retrieve attendance data from the backend and display it to users. To achieve a near real-time experience, the system can either periodically fetch updates from the backend or use a WebSocket-based approach where the backend pushes new attendance events to the frontend instantly when they are recorded. This allows the dashboard to update dynamically as employees check in or out.

In summary, due to the limitations of the fingerprint device, the system architecture requires a middleware component that continuously polls the device for attendance data and relays it to the backend. The backend handles storage, deduplication, and distribution of the data, while the frontend provides a real-time visualization of attendance activity. This design ensures compatibility with the existing hardware while still achieving a responsive and modern web-based user experience.
